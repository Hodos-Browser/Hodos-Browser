## TL;DR — what the adversarial pass changed

The research-grounded synthesis was then **refuted in six areas**, all source-verified. The corrections below are load-bearing:

1. **The multi-instance model was wrong (BLOCKER).** `SingleInstance` is **per-profile** (`SingleInstance.cpp:43`), and the **wallet is a process-wide singleton on fixed port 31301** shared by every open profile. There is no "previous session gone" cold-boot moment unless **every** instance is closed. Fix: an **ALL-INSTANCES-GONE global-mutex gate** (§D.0) before any apply; otherwise defer.
2. **ProfileLock is NOT released at browser-close (BLOCKER).** `ReleaseProfileLock` was deliberately moved into `main()` final cleanup after the DB-close cascade (`cef_browser_shell.cpp:553-558`) to avoid SQLITE_BUSY on fast relaunch. So `profile.lock` lingers into the installer+relaunch window. Fix: treat `profile.lock` as a first-class apply-time predicate + **widen the post-update relaunch retry** with an explicit no-browser-avoidance path (§D.2/§D.4).
3. **The wallet `/shutdown` is drain-based, not exit-based (BLOCKER).** `handlers.rs:193` only calls `shutdown.cancel()`; no `process::exit(0)` on the graceful path. A wallet mid-`createAction`/`broadcast` holds its image lock past `StopWalletServer`'s 5s wait -> `TerminateProcess` mid-SQLite-write = DB corruption. The job kill is a guarantee of **death, not clean WAL flush**. Raised as **OD-2** (money-process change, needs owner sign-off).
4. **Silent removes the OS "Run anyway" escape hatch (BLOCKER).** A validly-signed but SmartScreen/Smart-App-Control-blocked build silently bricks the fleet (the verify gates check **authenticity, not runnability**). Fix: **post-apply health gate + watchdog auto-revert + signer-continuity auto-degrade + server-side kill-switch/canary** (§D.6/§H.6/§H.7) — and **silent is disabled entirely until all four soak**.
5. **Mac coordination premise was false (BLOCKER).** `AutoUpdater_mac.mm` is **already shipped** with the old interface and a live `allowedChannelsForUpdater:@"beta"`. Model re-baselined to "Windows **extends** merged Mac code": `AutoUpdater.h`+`.mm` edited **atomically** with a macOS CI compile gate; `sparkle:channel` handled **oppositely per platform** (§G).
6. **Pragmatism (BLOCKER).** **0.4.0 ships notify-only**; the silent lock/rollback machinery is **deferred to 0.4.x**. The legacy migration is **corrected to `true->notify`** (v1's `true->silent` would have defaulted the whole installed base into the dangerous mode), and the silent option is **feature-flagged off** until its test gate + soak land.

**Net:** the safe half (notify, present-user Inno wizard) is a clean early deliverable; the silent apply-on-next-launch hybrid is the **end goal**, gated behind the all-instances-gone model, integrity-verified rollback, and the four fleet-safety mitigations.

## A. Architecture decision

**Chosen mechanism: (1) "shut children down → Inno apply on next launch → relaunch" — but ONLY for the SILENT mode, which 0.4.0 does NOT ship by default.** For the mode 0.4.0 actually ships (**notify**, default), the apply is the **present-user Inno wizard**, requiring none of the lock/rollback machinery below.

This split is the central correction folded in from the pragmatism review (Attack A/I) and the multi-instance/version-skew review (Attack A+D): **the high-risk apply-on-next-launch path exists to serve silent mode, and silent mode is deferred.** See §I for the descoped commit sequence and §L/OD-1 for the binding default decision.

**Why mechanism (1) is still the right shape for silent (when it ships in 0.4.x):**

- **Apply at the start of the next cold boot** means the previous session's children are *intended* to be gone — but this is **only true when the LAST instance of the entire app has closed**, not merely "this profile's session." The codebase contradicts the naive single-session assumption:
  - `SingleInstance::TryAcquireInstance(profileId)` is **per-profile** (pipe `\\.\pipe\hodos-browser-<profileId>`, `SingleInstance.cpp:43`), so **N concurrent `HodosBrowser.exe` processes** run when N profiles are open.
  - The wallet binds a **fixed global port 31301** and `LaunchWalletProcess()` *skips* launch if 31301 already listens — so the **first** profile owns the only `hodos-wallet.exe` (job-bound to it); other profiles reuse it over the port. The wallet is a **process-wide singleton shared by every instance**.
  - **Consequence:** there is no cold-boot moment where "the wallet is gone" unless **every** profile instance is simultaneously closed. Atomicity across the browser+wallet+adblock triple is therefore only achievable **when the last instance exits**. This is now a first-class precondition of apply (see §D.0 ALL-INSTANCES-GONE gate), not an assumption.
- **ProfileLock is NOT released at browser-close.** Verified at `cef_browser_shell.cpp:553-558`: `ReleaseProfileLock` was *deliberately moved out* of `ShutdownApplication` into `main()`'s **final cleanup**, after a DB-close cascade and after `CefShutdown`, precisely because a fast relaunch winning a prematurely-freed lock opened a live-WAL SQLite DB (`SQLITE_BUSY`). So `profile.lock` is held until very late in the *old* process's teardown — seconds that overlap the installer + relaunch window. This **invalidates** plan-v1's repeated "ProfileLock already gone at next launch" premise. `profile.lock` is now an apply-time predicate equal to the two child exes (§D.2), and the post-update relaunch has an explicit no-browser-avoidance failure path (§D.4).
- **The child-lifecycle infrastructure is real but only ~80% of what silent needs.** Both children are Job-Object-bound with `JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE` (`cef_browser_shell.cpp` ~3462 wallet / ~3653 adblock) and `StopWalletServer()`/`StopAdblockServer()` run in parallel (~543-550). **But the unsolved 20%** is: (a) the wallet `/shutdown` is **drain-based, not exit-based** — `handlers.rs:193` only calls `data.shutdown.cancel()`; the real exit is `server_handle.stop(true)` which *drains in-flight requests* with **no `process::exit(0)`** on the graceful path (contrast adblock, which force-exits 100ms after responding). A wallet mid-`createAction`/mid-`broadcast` holds `{app}\hodos-wallet.exe` locked until the drain completes, which can exceed `StopWalletServer`'s 5s `WaitForSingleObject` → `TerminateProcess` mid-SQLite-write = wallet-DB corruption. So **the job kill is a guarantee of *death*, NOT a clean WAL-flush guarantee.** This is reconciled in §E.1/§E.5.
- **Per-user / no-UAC preserved.** Inno stays `PrivilegesRequired=lowest`, `{localappdata}\HodosBrowser`, fixed `{app}` firewall rules, HKCU registry — mechanism (1) keeps all of it.

**Runner-up: (2) rename-and-replace (`MoveFileEx → .old`)** — **demoted to a logged-abort, NOT a corruption-causing fallback.** Plan-v1 used rename-fallback as the escape when the lock poll times out. The multi-instance review proved this **corrupts a running sibling**: renaming `hodos-wallet.exe` out from under a live Profile A, while Profile B writes a new wallet + relaunches on port 31301 still owned by A's old wallet, is exactly the version-skew/port-collision the plan claims to prevent. **New rule (§D.5):** if the lock is still held after the poll, **ABORT the apply, keep the old version, retry next launch** — rename is permitted **only** for a *provably-orphaned* child with **zero** live `HodosBrowser.exe` anywhere.

**Rejected: (3) Squirrel/Velopack versioned-folder + stub.** Unchanged from v1: Velopack's `current\`-swap shares our locked-file vulnerability (only `waitPid`s the main PID; our detached children block the rename like Velopack #4/#228); classic Squirrel's stub model breaks fixed-`{app}` firewall/`ProfileLock`/path assumptions and is unmaintained. Side-by-side staging + apply-on-restart is the right shape (which (1) has), not the right framework to import.

**Atomicity-model statement (folded in, required fix):** the update unit is the **shared singleton wallet on fixed port 31301**, which is process-wide across all profiles. An update can apply ONLY when the **last** instance has closed AND exclusive `GENERIC_WRITE` opens on all three exes succeed with zero retries. The "zero in-session version-skew" justification rests on the **all-instances-gone** gate (§D.0), never on a single-session assumption.

**C.1 — Where.** `%LOCALAPPDATA%\HodosBrowser\pending\` (per-user, **same volume as `{app}`** — required so the §D.5 *orphan-only* rename stays same-volume). Hodos-owned, mirrors electron-updater's `pending\` pattern.

## D. The apply-on-next-launch step (SILENT mode only; deferred to 0.4.x)

**D.0 — ALL-INSTANCES-GONE gate (NEW, the headline multi-instance fix).** Before any apply work, `MaybeApplyStagedUpdate()` must prove **no other `HodosBrowser.exe` is running for ANY profile** AND both children are fully terminated:
- Add a **GLOBAL (cross-profile) named mutex** `Global\HodosBrowser_AnyInstance` created by **every** `HodosBrowser.exe` instance at launch (the per-profile `SingleInstance` pipe **cannot see other profiles** and is therefore unusable for this). Cross-check by enumerating `HodosBrowser.exe` images (Toolhelp/`tasklist`).
- If **any** sibling instance is alive → **DEFER the apply** to a future launch and surface it (notify-mode prompt). **Never run the installer while a sibling holds the shared wallet.**
- **Gate behind `!g_picker_mode`** (new, folded): `MaybeApplyStagedUpdate` must not run in the picker process (`g_picker_mode`, ~3857) — the picker acquires no lock and launches no backends, and a picker applying updates while a separate profile holds the wallet is the same skew.

**D.1 — Exact startup sequencing in `cef_browser_shell.cpp`.** Insert `MaybeApplyStagedUpdate()` right after the `SingleInstance` check (~3907) and **before `AcquireProfileLock` (~3910)**, **before** `LaunchWalletProcess()`/`LaunchAdblockProcess()` (~3923–3924), and before window/overlay creation — **but only after D.0 passes**. Rationale holds *only* under the all-instances-gone precondition; without it the "previous session gone" claim is false.

**D.2 — Lock-release predicate = THREE exes + profile.lock, zero-retry, ABORT on timeout.** Confirm replaceability by attempting **exclusive `CreateFileA(..., GENERIC_WRITE, share=0)`** on:
- `{app}\hodos-wallet.exe`, `{app}\hodos-adblock.exe`, **and** (folded from Attack A/I DLL finding) the main `{app}\HodosBrowser.exe` and `libcef.dll` — all four are equally image-locked by any concurrent instance.
- `profile.lock` itself (folded from Attack D — it is held until the old process's late `main()` final cleanup; treat it as a first-class predicate).
- Do **not** use `FileExists` (the file exists while locked). Bounded poll (~10 s, `Sleep(250)`). **On timeout → ABORT-AND-RETRY-NEXT-LAUNCH** (§D.5), **never** proceed via rename-fallback when a sibling may be live. Because D.0 already proved zero siblings, a persistent lock here means a **truly-orphaned** child (the only case where §D.5 rename is permitted).

**D.3 — Run the staged installer.** With a verified marker + all locks free:
- Re-verify the marker's `sha256` against the staged file (tamper-while-staged defense).
- **Take the FULL `{app}` rollback backup FIRST** (folded from Attack A/B/I + Pragmatism): copy **all three exes + load-bearing DLLs/paks** (not just `HodosBrowser.exe`) to `pending\rollback\`, and write a content-hash manifest of the expected new file set for the §D.6 integrity check.
- Spawn the staged installer **detached**: `/VERYSILENT /SP- /SUPPRESSMSGBOXES /NORESTART`. `/NORESTART` because relaunch is the installer's `[Run]` job (§D.4). For **notify** mode this code path is **not used** — notify runs the present-user wizard (§F.5).
- After spawning, `MaybeApplyStagedUpdate()` calls **`exit(0)`** so this bootstrap holds nothing in `{app}` open. (This bootstrap has not yet acquired `profile.lock` — D.1 places apply before `AcquireProfileLock` — so `exit(0)` is correct for the bootstrap. The premise that matters is D.0: the *prior* session's `main()` final-cleanup must have fully completed, which the global mutex + exclusive `profile.lock` open in D.0/D.2 now actually verify.)

**D.4 — Relaunch — with an explicit no-browser-avoidance path (folded from Attack D, required fix).** Owned by Inno `[Run]` (`hodos-browser.iss:77`). **Remove `skipifsilent`** so a `/VERYSILENT` apply still relaunches. **New:** the relaunched `HodosBrowser.exe` MUST gate on `AcquireProfileLock` **succeeding**, and if it fails it must **NOT** show the "Profile is already in use" error-and-exit (current behavior, ~3910-3917). Because the dying process holds `profile.lock` until its late final-cleanup, the existing **6×500ms = 3s** retry is likely insufficient for a post-update relaunch. **Fix (code change gated to the post-update relaunch case):** widen the ProfileLock retry window specifically for the post-update relaunch (e.g. detect a "just-applied" marker → extend retry to a bounded longer window while polling `profile.lock` with exclusive `CreateFile`), so a user-initiated update **never** ends in no-browser.

**D.5 — Failure handling / rollback (fail-closed, keep old version).**
- **Lock still held after D.2 poll, sibling proven gone in D.0 → truly-orphaned child:** same-volume `MoveFileEx(child.exe → child.exe.old, MOVEFILE_REPLACE_EXISTING)` for the orphaned child only; schedule `.old` deletion next launch. **This is the ONLY case rename is permitted.** If D.0 could not prove zero siblings, the path never reaches here — it ABORTS and retries next launch (no rename, no corruption). `MOVEFILE_DELAY_UNTIL_REBOOT` remains **forbidden** (needs HKLM/admin; breaks per-user/no-UAC).
- **Installer fails / interrupted (power loss mid-swap):** because apply runs before `AcquireProfileLock` and before the old session does anything, the worst case is partially-replaced binaries. Mitigated by the **full `{app}` rollback backup** (§D.3) + the §D.6 integrity check. If apply can't complete, **keep running the old version and retry next launch** — never leave the user with no browser.
- **Orphaned-child instrumentation (new, folded from Attack D):** log + telemeter which path actually fired (graceful-free / orphan-rename / abort-retry) so soak testing distinguishes a *healthy-but-slow* shutdown from a *truly-orphaned* child. Without this the poll could silently rename/kill on a slow-but-healthy shutdown and risk DB corruption.
- **On success (= healthy marker written, §D.6):** delete `pending\` including `rollback\`.

**D.6 — POST-APPLY LAUNCH-SUCCESS GATE + integrity verification (NEW, folded from Attack A/B/I — closes the silent-brick).** Plan-v1 discarded `rollback\` on installer exit 0, which says nothing about whether the new binaries *run* under current OS policy (SmartScreen/Smart App Control/AV can validly block a correctly-signed build). **Fix:**
- After the `/VERYSILENT` relaunch, the new `HodosBrowser.exe` writes a **`first-run-healthy` marker** ONLY after it (a) acquired `profile.lock` AND (b) both children passed health checks on 31301/31302.
- `pending\rollback\` is **retained until that marker exists.** If the next launch finds an **armed-but-unconfirmed** apply (no healthy marker), it **auto-restores from `rollback\` and pins to the old version**, converting "silent brick" → "auto-recover + retry."
- **Broken-tree predicate verifies INTEGRITY, not presence** (folded from Attack A+D + Pragmatism): validate **every** expected exe/DLL by **Authenticode signature AND content-hash against the §D.3 stage-time manifest** — `build-release.ps1`'s `$requiredFiles` *presence* check is insufficient because `ignoreversion` copies can leave a **present-but-truncated** binary after power loss. Rollback triggers on present-but-unrunnable (wrong version / bad signature / failed port-bind / crash-loop), not just missing files.
- **Watchdog auto-revert:** if the just-applied build fails to produce the healthy marker within N seconds across M consecutive launches, restore `rollback\`, set `updateMode` effectively to **paused**, and surface a notification.

## E. Child-process shutdown + lock release

**E.1 — Existing infrastructure (verified) and its honest limits.**

**E.2 — Cross-profile AppMutex + SetupMutex (reconciled with multi-instance, folded from Attack A+D).**
- `HodosBrowser.exe` creates `Global\HodosBrowser_AnyInstance` at launch (this is the **same global mutex** as §D.0, doubling as the Inno AppMutex target).
- Add to `[Setup]` in `hodos-browser.iss`: `AppMutex=...` (detect a live browser) and `SetupMutex=HodosBrowserSetup` (prevent two installers racing).

**E.3 — Installer-side defensive shutdown (Inno `[Code]`) — bounded, no hang.** Safety net for a manually-run installer or orphaned child:

- **Last-resort `taskkill /F /IM hodos-wallet.exe /IM hodos-adblock.exe`** by image name ONLY after the graceful POST + bounded poll time out. Comment that hard-kill mid-SQLite-write is **accepted only as last resort** (Inno's Restart Manager *detects* GUI-less console children via `NtQueryInformationFile` but cannot gracefully stop them — returns `ERROR_FAIL_SHUTDOWN`). Prefer `CloseApplications=force` semantics **only** as this final fallback.
- **Do NOT** call `RegisterApplicationRestart` in the children — only `HodosBrowser.exe` is the relaunch target (§D.4), and an Inno-relaunched backend would be orphaned, not job-bound.

**E.4 — Interaction with ProfileLock + relaunch (reconciled).** Apply runs before `AcquireProfileLock`; the installer's `/NORESTART` + `[Run]` relaunch starts a fresh process. **But** because `profile.lock` releases only at the old process's late final-cleanup, the relaunch gates on the **widened** ProfileLock retry (§D.4) and on the same exclusive-open/lock-free predicate (§D.2), with the explicit no-browser-avoidance fallback. - **Signer-continuity gate:** record the currently-installed build's Authenticode signer + cert thumbprint (`signerThumbprint` in the §C.3 marker); **auto-degrade silent → notify** when the staged build's signer/thumbprint differs, forcing a user-visible "Run anyway"-equivalent confirmation during exactly the reputation-reset window.
- The §D.6 post-apply health gate + watchdog auto-revert convert an OS-blocked silent apply from "brick" to "auto-recover."
- These four (health gate, watchdog, kill-switch §H.7, signer-continuity) make **OD-1 binding** (§L): silent is **disabled entirely** until all four ship and soak.

**H.7 — Server-side kill-switch + staged/canary rollout (NEW, folded from Attack A/B/I).** Plan-v1 had no mechanism to stop one bad feed entry hitting 100% of the fleet in one poll. **Add to the feed design:** percentage-gated rollout, a **retractable** feed entry, and a minimum canary-dwell before public. Combined with the signer-continuity gate, a single bad appcast item cannot reach the whole fleet in one cycle. This is a prerequisite for enabling silent (OD-1).

## J. Test plan

Each case: action → pass criteria → parity note (how the macOS Sparkle path checks the same property).

1. **Happy path (silent, 0.4.x).** Stage a verified newer installer; **ensure zero sibling instances**; close; relaunch. **Pass:** D.0 confirms no sibling; `MaybeApplyStagedUpdate` runs `/VERYSILENT` before ProfileLock; relaunch acquires `profile.lock` via the widened retry; **post-apply healthy marker written**; `pending\`+`rollback\` cleaned; gold pill + wallet send/receive work post-update. **Parity:** macOS Sparkle silent-on-quit (SU* keys) lands the new `.app`; verify `CFBundleVersion` bumped.
2. **Deferred apply while a SECOND profile is running (NEW — the case that breaks plan-v1).** Open Profile A and Profile B; close B; reopen B with a verified stage present. **Pass:** D.0 detects A alive → **apply DEFERS**, no installer runs, B starts normally on the shared old wallet; apply succeeds only after A also closes. **No rename-fallback fires while A is live; no port-31301 collision; no version-skew.** **Parity:** macOS replaces the `.app` wholesale; note Windows-specific shared-singleton-wallet model.
3. **Locked-file apply (truly-orphaned child).** Orphan a `hodos-wallet.exe` with **zero** live `HodosBrowser.exe`. **Pass:** D.0 passes (no siblings); D.2 poll detects lock; graceful `/shutdown` POST + bounded wait; if still locked, **orphan-only** rename (`MoveFileEx → .old`) lets Inno write; `.old` cleaned next launch; telemetry records "orphan-rename"; **wallet DB not corrupted.** **Parity:** macOS has no detached-image-lock issue — Windows-specific.
4. **Version-downgrade / rollback rejection — INTEGER comparator.** Point appcast at `build-number ≤ current`, including a **lexically-higher-but-older** crafted `sparkle:version` (e.g. `0.4.10` vs `0.4.2`). **Pass:** integer anti-rollback (§B.4 #3 / §H.4) refuses to stage; no marker. **Parity:** macOS `SUStandardVersionComparator` + `--build-number` rejects.
5. **Tampered / unsigned installer + stale-replayed appcast rejection.** Corrupt the staged exe; strip Authenticode/EdDSA; feed a CVE-2013-3900 padded-cert sample; **replay an old-but-validly-signed appcast+installer**. **Pass:** EdDSA OR `WinVerifyTrust`(+`EnableCertPaddingCheck`) fails → reject + delete + fail-closed; **the appcast-document signature / monotonic feed-version (§B.4 #4) rejects the replay** even though per-enclosure sigs verify. **Parity:** macOS Sparkle EdDSA rejects a tampered DMG.
6. **Client fail-closed init (NEW).** Ship a build whose embedded EdDSA/DSA key fails to parse. **Pass:** `win_sparkle_init` is NOT called; updater is **off**; browser runs normally; no unverified enclosure ever accepted (§B.2a). **Parity:** macOS aborts Sparkle init on missing `SUPublicEDKey`.
7. **Interrupted apply (power loss mid-swap) — TRUNCATED binary, not missing.** Kill the installer mid-`/VERYSILENT`, leaving a **present-but-truncated** DLL/exe. **Pass:** next startup's **integrity check (Authenticode + content-hash manifest, §D.6)** detects the corrupt-but-present binary (a mere `$requiredFiles` presence check would NOT), restores from `pending\rollback\`, runs old version, retries — **never "no browser."** **Parity:** macOS atomic bundle swap is harder to half-apply; Windows needs the explicit integrity-verified rollback.
8. **Post-apply OS-block / SmartScreen-flagged build under silent (NEW — silent-brick closure).** Apply a correctly-signed build that Smart App Control/AV blocks at launch. **Pass:** the **post-apply healthy marker is never written** → next launch **auto-restores rollback + pins old version**; watchdog sets mode to paused + notifies; **signer-continuity gate** had already auto-degraded silent→notify if the signer/thumbprint changed. **Parity:** macOS notarization+stapling is the equivalent gate — verify the DMG is notarized.
9. **ProfileLock relaunch race (no-browser-avoidance).** Force the old process's `profile.lock` to linger into the relaunch window. **Pass:** the **widened** post-update ProfileLock retry (D.4) + exclusive-open gate succeed; the relaunch does **NOT** hit the "Profile in use" error-and-exit; single instance does not silently die. **Parity:** macOS single-instance via `flock` — verify no double-launch.
10. **Dev-vs-prod isolation untouched.** Run the updater path under `HODOS_DEV=1`. **Pass:** download/stage/apply **inert**; prod data dir untouched; the `target/release`/`build/bin/Release` dev-detection safeguard still refuses bare launch. **Parity:** macOS dev guard (`HODOS_DEV=1`, `HodosBrowserDev/`) equally inert.
11. **Dual-signed appcast verified by a SIMULATED real 0.8.1 client (BLOCKING gate with commit 3b).** Generate a dual-signed appcast; point a **real WinSparkle 0.8.1** (DSA-only) build AND a 0.9.3 (EdDSA) build at it. **Pass:** 0.8.1 validates `sparkle:dsaSignature` + offers the update; 0.9.3 validates `sparkle:edSignature` (ignores DSA per header `@note`); **neither population stranded.** Plus the **§G coordination smoke gate** asserts the same XML still carries the macOS item's `edSignature` + channel. **Parity:** macOS Sparkle EdDSA is already the only path — verify the macOS item still validates after the generate-appcast.py change.
