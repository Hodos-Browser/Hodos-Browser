## Cluster A — Sigma protocol, Sigma Identity, BAP, BMAP

### A1. Dependency graph

```
On-chain primitives:
  bitcoinschema/sigma         on-chain SIGMA OP_RETURN signing (active, last push 2026-02-06, 5 stars)
  bitcoinschema/bap           Bitcoin Attestation Protocol identity (active, last push 2026-03-18, 0 stars)

Web/auth product (b-open-io org, run by rohenaz / Satchmo):
  b-open-io/bitcoin-auth      X-Auth-Token signer/verifier lib; the cryptographic core (last push 2026-03-10, 9 stars)
  b-open-io/better-auth-plugin   Sigma's official Better Auth integration (last push 2026-04-30, very active)
  sigma-auth (the hosted service)  auth.sigmaidentity.com OAuth 2.1/OIDC server. The repo URL the marketing site
                                   advertises (github.com/b-open-io/sigma-auth) returns 404 — either private or renamed.

Indexer / data layer:
  rohenaz/bmap-api            ARCHIVED 2025-01-30 (11 stars). Replaced by go-junglebus and 1sat-stack within b-open-io.
```

**Relationships:**

- `sigma` is the on-chain layer: appends `SIGMA <algorithm> <address> <signature> <vin>` to a tx output, where `algorithm` is `BSM` or `BRC-77`, signing the spent input's txid + the locking script.
- `bap` is identity: an OP_RETURN protocol with `1BAPSuaPnfGnSBM3GLV9yhxUdYe4vGbdMT` prefix and actions `ID / ATTEST / ALIAS / DATA / REVOKE`. AIP (Author Identity Protocol) signs the payload.
- `bitcoin-auth` is the request-signing primitive used by Sigma Identity for HTTP-level auth. It is **not** on-chain — it is per-request signed headers.
- `Sigma Identity` (the OAuth product at `auth.sigmaidentity.com`) is a hosted server that **issues OAuth 2.1/OIDC tokens** after verifying a `bitcoin-auth`-format signature, and **reads BAP profile data** from the chain (or a BMAP-style indexer) to enrich `/userinfo` responses.
- `BMAP` (parser) is dead in its archived form. Functional replacement: 1sat-stack / go-junglebus pipelines.

### A2. What bytes does the user sign?

Per `b-open-io/bitcoin-auth` README (canonical implementation):

**Token format:** `pubkey|scheme|timestamp|requestPath|signature`

**Signed message body:** an ASCII concatenation of:
- `requestPath` (with query string)
- ISO8601 `timestamp`
- `SHA256(body)` if a body is present
- the literal `scheme` identifier (`bsm` or `brc77`)

**Default scheme: `brc77`** (BRC-77 / `SignedMessage.sign()` from `@bsv/sdk`, derived child keys, signer pubkey embedded).
**Legacy: `bsm`** (Bitcoin Signed Message, magic-prefix SHA-256 ECDSA).

A 5-minute time-skew window is enforced (`timePad` default 5 min). Signatures are request-specific: same key + different path = different signature, so they're not replayable across endpoints.

> Source: https://github.com/b-open-io/bitcoin-auth (README, last push 2026-03-10)

### A3. OAuth flow at the HTTP level

From `sigmaidentity.com/docs/reference/api`:

```
GET /oauth2/authorize
    ?client_id=…
    &redirect_uri=…
    &response_type=code
    &provider=sigma|google|github   (optional)
    &state=…
    &scope=…
    &code_challenge=…&code_challenge_method=S256   (PKCE)

POST /api/auth/oauth2/token       (code → access_token, ES256-signed JWT, 30-day TTL, no refresh)
GET  /api/auth/oauth2/userinfo    (returns OIDC claims + pubkey + bap{idKey,rootAddress,currentAddress,…})

POST /sigma/authorize             (direct Bitcoin auth — bypasses interactive flow,
                                   client signs with bitcoin-auth and posts the token)
```

Token response shape (verbatim from docs):
```json
{
  "access_token": "eyJhbGciOiJFUzI1NiIs...",
  "token_type":   "Bearer",
  "expires_in":   2592000
}
```

OpenAPI spec exists at `https://auth.sigmaidentity.com/api/auth/reference` (Scalar UI; needs browser render to extract).

### A4. Critical question — does Sigma accept any BAP key, or only its own?

**Answer: it accepts any valid signature from any keypair, and BAP profile lookup is agnostic to key origin.**

From `sigmaidentity.com/docs/reference/bap-profiles`, verbatim:
> "Sigma Auth automatically fetches and includes BAP profile data when available… It simply needs a valid `idKey` to fetch associated profile data from the blockchain."
> "Sigma Auth acts as a consumer of BAP data rather than a gatekeeper."

The verifier function (`verifyAuthToken` in `bitcoin-auth`) only checks: signature ↔ pubkey ↔ message. It does not check that the key was generated by Sigma's own session-keypair flow. **This means Hodos can in principle present a wallet-rooted key, sign the `/sigma/authorize` request with it, and Sigma will accept the auth** — provided the BAP `idKey` resolves on-chain.

**Caveats this report cannot verify from public docs:**
- Whether the hosted server enforces any extra "must have been issued by us" check at the cookie/session layer that the public docs don't mention
- Whether `/sigma/authorize` accepts arbitrary BAP idKeys never previously seen at `social.sigmaidentity.com`, or whether it requires a profile registration first
- Whether `pubkey` recovered from the signature is matched against any allow-list

**These need a mitmproxy capture against a live working integration. Don't ship Hodos's interception logic on the assumption of permissiveness without empirical proof.** BSVradar.com is now a confirmed Sigma user — primary mitmproxy target.

### A5. Is there a `window.sigma.*` global?

**No.** Sigma Identity is a **server-side OAuth provider**, not a content-script wallet bridge. Integration is via `authClient.signIn.sigma({ clientId, … })` from `@sigma-auth/better-auth-plugin` (server-side SDK), which redirects the browser to `auth.sigmaidentity.com/oauth2/authorize`. The user's signing happens *inside that domain's pages* using an in-page bundled keypair (the ephemeral browser keypair). There is no app-injected `window.sigma` for dApps to call.

This is exactly why the "intercept and substitute Hodos's key" approach is the right one — there's no extension API to override. Hodos would intercept the redirect to `auth.sigmaidentity.com` and have the wallet sign the challenge instead of the in-page ephemeral keypair.

### What this means for Hodos (Cluster A)

Sigma's auth is just `bitcoin-auth` (BRC-77) signatures over OAuth, validated against BAP-anchored identities. No wallet API exists, so the Hodos browser must intercept the redirect to `auth.sigmaidentity.com` and either (a) drive the in-page UI with a synthetic keypair derived from the Hodos wallet (fragile), or (b) intercept the `POST /sigma/authorize` call and sign it with Hodos's BRC-77 child key directly. Option (b) is cleaner but requires verifying empirically that the server doesn't gate by "session was initiated through our UI" — capture a real flow with mitmproxy before committing.

## Cluster D — Sigma vs. 1Sat Ordinals: how related?

**They are independent at the protocol layer, partially overlapping at the identity layer, and tightly aligned at the human/team layer.**

- **Protocol independence:** 1Sat Ordinals are inscriptions in P2PKH-controlled UTXOs (BSV20/21 fungibles, NFTs as 1-sat outputs). Ownership = key control of the output's address. Sigma Identity is OAuth over signed HTTP requests. **Owning a 1Sat ordinal does not log you into anything; logging in via Sigma does not move ordinals.**
- **Shared crypto layer:** Both ultimately use secp256k1 keys. If a user's BAP `idKey` (Sigma identity root) and their ordinal-owning address are derived from the same BIP32 master, then the same wallet that signs Sigma's `/sigma/authorize` token also signs the spend of their ordinals. This is *de facto* true for users of Yours Wallet, since Yours derives both from one mnemonic. But Sigma Identity itself does not "see" the user's ordinals — `/userinfo` returns BAP profile fields, not ordinal inventory.
- **Team / repo layer:** Both come out of the same orbit — Luke Rohenaz (rohenaz / Satchmo) leads `b-open-io` which ships Sigma Auth, the better-auth plugin, 1sat-sdk, 1sat-wallet-toolbox, ordfs-server, and `1satordinals.com`. The Sigma + 1Sat integration story is "same author, same SDK family" rather than "same protocol."
- **How a Sigma session would access a user's ordinal:** It wouldn't, on its own. The app would need to call `userinfo`, get the BAP `currentAddress`, and **separately** query an indexer (1sat-stack, JungleBus) for ordinals at that address.

### What this means for Hodos (Cluster D)

For Hodos's auth-substitution work, treat Sigma and 1Sat as separate concerns. **Sigma interception is purely about HTTP-level identity** — substituting which key signs `/sigma/authorize`. **1Sat ordinal payment flow is independent.** The one connection point worth designing for: when Hodos signs Sigma auth with a wallet-rooted key, that same key's `currentAddress` should be the one holding the user's ordinals, so Sigma's `/userinfo` BAP block surfaces an address the dApp can immediately query for the user's 1Sat inventory. Make sure the Hodos wallet's BAP-identity-key derivation path lines up with the address it would use for ordinal receipt.

1. Does `auth.sigmaidentity.com` accept a `/sigma/authorize` POST signed by a key that has **never** registered at `social.sigmaidentity.com`? → **mitmproxy a real flow on BSVradar.com**.
